# 001 — Repo decomposition and naming

**Decision.** Implementations live in sibling repos named `isnow.<lang>` (this repo's Makefile already names that convention): `tsvsheet/isnow.go` (Go) and `tsvsheet/isnow.js` (JS). The Go repo is a single repo carrying both the importable library (root package `isnow`, the nats.go naming precedent) and the CLI (`cmd/isnow`, binary `isnow`, goreleaser `project_name: isnow`). The playground lives in the existing `www.uplang.org` under `/isnow/`. Docs: `docs.isnow.go`, `docs.isnow.js`.

**Alternatives.** (a) A separate CLI repo (`isnow-cli`) mirroring the library/CLI class split — rejected: it would ship a near-empty repo whose only content is command wiring, and the org's own CLI precedent (`up-sh`) keeps wiring and logic together; (b) renaming this repo to `isnow.grammar` to free `isnow` for the CLI — rejected: the user deliberately named the grammar home `isnow`, and a rename is their call, not ours; (c) full implementations in five languages — rejected (YAGNI): `make gen` keeps python/java/cpp parsers one command away, and the corpus makes future implementations cheap.

**Consequence.** `isnow.go` classifies as a CLI-class repo (go.mod + `cmd/isnow`); it carries no `library.go` marker to avoid a dual-class conflict — the root package is real library code, importable as `github.com/tsvsheet/isnow.go`. The binary name (`isnow`) intentionally differs from the repo name; `.goreleaser.yml` pins `project_name: isnow` and any layout-oracle complaint is recorded in `.standards.yaml` with this ADR as the reason.
